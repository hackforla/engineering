<h4><a id="user-content-step-1-become-a-member-of-the-repository-team" class="anchor" aria-hidden="true" href="#step-1-become-a-member-of-the-repository-team"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Step 1: Become a member of the repository Team</h4>

</li>
<li>

<p>In <a href="https://github.com/hackforla/website">https://github.com/hackforla/website</a>, look for the fork icon in the top right. Click it and create a fork of the repository.</p>
<p>For git beginners, a fork is a copy of the repository that will be placed on your GitHub account url.</p>

<h4><a id="user-content-step-3-clone-your-online-repository-to-your-local-computer" class="anchor" aria-hidden="true" href="#step-3-clone-your-online-repository-to-your-local-computer"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Step 3: Clone your online repository to your local computer</h4>
<p>For git beginners, this process will create a third copy of the repository on your local desktop.</p>
<p>First create a new folder on your desktop that will contain <code>hackforla</code> projects.</p>
<p>In your shell, navigate there then run the following commands:</p>
<div class="highlight highlight-source-shell"><pre>git clone https://github.com/your_GitHub_user_name/website.git</pre></div>
<p>You should now have a new folder in your <code>hackforla</code> folder called <code>website</code>. Verify this by changing into the new directory:</p>
<div class="highlight highlight-source-shell"><pre><span class="pl-c1">cd</span> website</pre></div>
<p>Next, verify that your local cloned repository is pointing to the correct <code>origin</code> URL (that is, the forked repo on your own Github account):</p>
<div class="highlight highlight-source-shell"><pre>git remote -v</pre></div>
<p>You should see <code>fetch</code> and <code>push</code> URLs with links to your forked repository under your account (i.e. <code>https://github.com/YOURUSERNAME/website.git</code>). You are all set to make working changes to the website on your local machine.</p>
<p>However, we still need a way to keep our local repo up to date with the deployed website. To do so, you must add an upstream remote to incorporate changes made while you are working on your local repo. Run the following to add an upstream remote URL &amp; update your local repo with recent changes to the <code>hackforla</code> version:</p>
<div class="highlight highlight-source-shell"><pre>git remote add upstream https://github.com/hackforla/website.git
git fetch upstream</pre></div>
<p>After adding the upstream remote, you should now see it if you again run <code>git remote -v</code> :</p>
<div class="highlight highlight-source-shell"><pre>origin  https://github.com/YOURUSERNAME/website.git (fetch)
origin  https://github.com/YOURUSERNAME/website.git (push)
upstream        https://github.com/hackforla/website.git (fetch)
upstream        https://github.com/hackforla/website.git (push)
</pre></div>
<p>If you accidentally cloned using the repository URL from the HackForLA Github (instead of the fork on your Github), then you can correct that with the following two commands:</p>
<ol>
<li>Set your forked repo on your Github as an <code>origin</code> remote:</li>
</ol>
<div class="highlight highlight-source-shell"><pre>git remote set-url origin https://github.com/your_user_name/website.git</pre></div>
<ol start="2">
<li>Add another remote called <code>upstream</code> that points to the <code>hackforla</code> version of the repository. This will allow you to incorporate changes later:</li>
</ol>
<div class="highlight highlight-source-shell"><pre>git remote add upstream https://github.com/hackforla/website.git</pre></div>

<h4><a id="user-content-step-6-work-on-an-issue-using-git" class="anchor" aria-hidden="true" href="#step-6-work-on-an-issue-using-git"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Step 6: Work on an issue using git</h4>
<p>Create a new branch for each issue you work on. Doing all your work on topic branches leaves your repository's main branch (named <code>gh-pages</code>) unmodified and greatly simplifies keeping your fork in sync with the main project.</p>
<p>a) Check current branch</p>
<p>The <code>git branch</code> command will let you know what branch you are in, and what branch names are already in use.</p>
<div class="highlight highlight-source-shell"><pre>git branch</pre></div>
<p>You will see a list of all of your branches. There will be a star (<code>*</code>) next to the branch that you are currently in. By default you should start on the <code>gh-pages</code> branch.</p>
<p>Note: when you work on future issues, you must always be in the <code>gh-pages</code> branch when creating a new branch.</p>
<p>If you are not currently in the <code>gh-pages</code> branch, run the following command to return to it:</p>
<div class="highlight highlight-source-shell"><pre>git checkout gh-pages</pre></div>
<p>b) Create a new branch where you will work on your issue</p>
<p>The <code>git checkout</code> command will create and change to a new branch where you will do the work on your issue.  In git, the checkout command lets you navigate between different branches.  Using the <code>-b</code> flag you can create a new branch and immediately switch into it.</p>
<p>To create a new issue branch, and switch into it:</p>
<div class="highlight highlight-source-shell"><pre>git checkout -b fix-logo-width-311</pre></div>
<p>The text after the <code>-b</code>, in the example <code>fix-logo-width-311</code>, will be the name of your new branch. Choose a branch name that relates to the issue you're working on. (No spaces!)</p>
<p>The format should look like the scheme above where the words are a brief description of the issue that will make sense at a glance to someone unfamiliar with the issue.</p>
<p>No law of physics will break if you don't adhere to this scheme, but laws of git will break if you add spaces.</p>
<p>When you've finished working on your issue, follow the steps below to prepare your changes to push to your repository.</p>
<p>c) Prepare your changes to push to your repository</p>
<p>Once you are done with the work on your issue you will push it to your repository.  Before you can push your work to your repository, you will stage and commit your changes.  These two commands are similar to the save command that you have used to in other programs.</p>
<blockquote>
<ul>
<li>If you are using Visual studios code you can use the Git graphical user interface to stage your changes. For instructions check out the [Git gui wiki] (<a href="https://github.com/hackforla/website/wiki/Using-Git-GUI-(Graphical-user-Interface)-in-Visual-Studios-Code">https://github.com/hackforla/website/wiki/Using-Git-GUI-(Graphical-user-Interface)-in-Visual-Studios-Code</a>)
Alternatively you can follow the intstructions below to stage changes through the terminal.</li>
</ul>
</blockquote>
<p>-Use the <code>git add</code> command to stage your changes.<br>
This command prepares your changes before you commit them. You can stage files one at a time using the filename.</p>
<p>Run the command:</p>
<div class="highlight highlight-source-shell"><pre>git add “filename.ext”</pre></div>
<p>-Use the <code>git status</code> command to see what files are staged.</p>
<p>This command will list the files that have been staged.  These are the files that will be committed (saved) when you run the next command, <code>git commit</code>. Please be sure all your staged changes are relevant to the issue you are working on. If you find you have included unrelated changes, please unstage them before making this commit - and then make a new commit for the unrelated changes. (The commands for unstaging commits are provided in the output of your <code>git status</code> command.)</p>
<div class="highlight highlight-source-shell"><pre>git status</pre></div>
<p>-Use the <code>git reset HEAD</code> command to remove a staged file.</p>
<p>This command will remove a file that has been staged.  This file will not be committed (saved) when you run the next command, <code>git commit</code>. This only works if the wrong files were added, but they were not yet committed. The file will be removed from the staging area, but not actually deleted:</p>
<div class="highlight highlight-source-shell"><pre>git reset HEAD “filename.ext” </pre></div>
<p>-Use the <code>git commit</code> command</p>
<p>This command saves your work, and prepares it to push to your repository.  Use the <code>-m</code> flag to quickly add a message to your commit. Your message should be a short description of the issue you are working.  It will be extremely helpful if other people can understand your message, so try to reisst the temptation to be overly cryptic.</p>
<p>To commit your changes with a message, run:</p>
<div class="highlight highlight-source-shell"><pre>git commit -m “insert message here”</pre></div>
<p>Congratulations!  You are now ready to push your work to your repository.</p>

<div class="highlight highlight-source-shell"><pre>git fetch upstream</pre></div>

<p>If you do not see any output, there have not been any changes in the
main Hack for LA website repository since the last time you

<p>If you just type <code>git push</code> you will be prompted to create a new branch in your GitHub repository. The name at the end of this command should be the same as the name of the local branch that you created back in step 6, as in the example below:</p>
<div class="highlight highlight-source-shell"><pre>git push --set-upstream origin fix-logo-width-311</pre></div>

<p>Your fork of this repository on GitHub, and your local clone of that fork, will
get out of sync with this (upstream) repository from time to time.  (That's what has happend when you see something like "This branch is 1 commit behind hackforla:gh-pages" on the github website version of your hackforla repository.)</p>
<p>One way to keep your fork up to date with this repository is to follow
these instruction: <a href="https://github.com/KirstieJane/STEMMRoleModels/wiki/Syncing-your-fork-to-the-original-repository-via-the-browser">Syncing your fork to the original repository via the browser</a></p>
<p>You can also update your fork via the local clone of your fork, using
these instructions. Assuming you have a local clone with remotes
<code>upstream</code> (this repo) and <code>origin</code> (your GitHub fork of this repo):</p>
<p>First, you will need to create a local branch which tracks upstream/gh-pages.  You will only need to do this once; you do not need to do this every time you want to incorporate upstream changes.</p>
<p>Run the following two commands:</p>
<div class="highlight highlight-source-shell"><pre>git fetch upstream
git checkout -b upstream-gh-pages --track upstream/gh-pages</pre></div>
<p>If you have already created the branch upstream-gh-pages, the following commands will incorporate upstream changes:</p>
<div class="highlight highlight-source-shell"><pre>git checkout upstream-gh-pages <span class="pl-c"><span class="pl-c">#</span> Move to the branch you want to merge with. </span>
git pull  <span class="pl-c"><span class="pl-c">#</span> This updates your tracking branch to match the gh-pages branch in this repository</span>
git checkout gh-pages  <span class="pl-c"><span class="pl-c">#</span> Move back to your gh-pages branch</span>
git merge upstream-gh-pages  <span class="pl-c"><span class="pl-c">#</span> Merge to bring your gh-pages current. </span></pre></div>
<p>If you do all your work on topic branches (as suggested above) and keep gh-pages free of local modifications, this merge should apply cleanly.</p>
<p>Then push the merge changes to your GitHub fork:</p>
<div class="highlight highlight-source-shell"><pre>git push</pre></div>
<p>If you go to your online github repository this should remove the message "This branch is x commit behind hackforla:gh-pages".</p>

<p>To incorporate these updates from the main GitHub repository into your
topic branch, you can 'rebase' your branch onto your updated gh-pages
branch. NOTE you should only rebase if you have never pushed your
topic branch to GitHub (or shared it with another collaborator).</p>
<div class="highlight highlight-source-shell"><pre>git checkout fix-logo-width-311
git rebase gh-pages</pre></div>
<p>If you receive warnings about conflicts, abort the rebase with <code>git rebase --abort</code> and instead merge gh-pages into your branch.</p>
<div class="highlight highlight-source-shell"><pre>git checkout fix-logo-width-311
git merge gh-pages</pre></div>

<p>If you have not submitted a pull request make sure to write a weekly status update on your issue before the Sunday meeting. Follow the format below and add pictures of any visual changes made to the site.</p>
<ol>
<li>Progress: "What is the current status of your project? What have you completed and what is left to do?"</li>

<div class="highlight highlight-source-shell"><pre>git push --set-upstream origin fix-logo-width-311</pre></div>
<p>Now create a new pull request to ask for your updates to be
incorporated into the live web site. Go to
<a href="https://github.com/hackforla/website/pulls">https://github.com/hackforla/website/pulls</a> and click on "New pull
request". Please rename your pull request something descriptive i.e. "building a project card for civic opportunity project".
Also, since your changes are not in the hackforla/website
repository, you need to click the "compare across forks" link in the
first paragraph to make you repository and your new branch
available. Make sure to include pictures of any visual changes made to the site and document your edits on the pull request so that the reviewer can understand the changes made. Review the changes that will be included in the pull
request and, if it fixes a specific issue, include <code>Fixes #140</code> in the
pull request message so the issue will be closed automatically once
your pull request is accepted and merged.</p>
<p>Once you have finished working on the issue you have chosen, commit
the changes to your local branch (e.g. <code>fix-logo-width-311</code>).</p>
<p>Important: After you completed your assignment and committed all of the changes, before moving onto your next issue and creating a new branch, you must leave your current branch and return to the <code>gh-pages</code> branch. From there you can checkout into a new branch. (This ensures you don’t accidentally include the changes from your previous branch in your new branch).</p>
<p>Run the following command to return to the <code>gh-pages</code> branch:</p>
<div class="highlight highlight-source-shell"><pre>git checkout gh-pages</pre></div>
<p>From here, once your pull request is approved and merged you can pull the recent merge from the Hack For LA repository and delete your local branch:</p>
<div class="highlight highlight-source-shell"><pre>git pull upstream gh-pages
git branch -d <span class="pl-k">&lt;</span>your-feature-branch<span class="pl-k">&gt;</span></pre></div>
<p>Managing branches this way will keep the commit logs cleaner on the Hack For LA repository, versus merging your completed feature branches into your local repo.</p>
<p>Now you are all set to work on a new PR. Start over on Step 6.</p>
